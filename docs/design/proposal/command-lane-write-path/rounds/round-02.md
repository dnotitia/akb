# Round 02 — 2026-06-10 — PM 검증 요청 2건 반영

## 1. "git↔PG가 이미 transactional하다는 이야기" 검증 → §1.3 수정

kwoo24-oss의 design 외 최근 작업을 추적해 확인:

- `553484b` fix(0.3.7) #99 — writer당 커넥션 1개 + `_path_lock` 단일
  트랜잭션. **PG 내부 원자성은 사실** (row+chunks+relations+events 일괄
  롤백). "transactional 처리" 이야기의 실체로 추정.
- `12c3ba9` fix(0.8.6) #170 (2026-06-08) — `akb_get`이 본문을 floating
  HEAD가 아니라 **PG row의 `current_commit`에 핀**해서 읽도록 변경.
  git이 PG보다 앞서도 사용자에게 노출되지 않게 됨. 초안 §1.3의 "검색과
  본문 불일치" 증상 서술은 이 수정으로 낡은 정보가 됨 → 정정.
- 단, put(:318→:343)/update(:614→:622) 모두 git 커밋이 PG tx 중간에
  실행되고 **PG 실패 시 git ref를 되돌리는 보상 코드는 없음**
  (`resource_integrity.py`는 hash projection 수리 도구일 뿐). 크로스
  스토어 원자성 부재 자체는 여전히 사실 — 피해 범위가 "서빙 불일치"에서
  "히스토리 오염(스트레이 커밋)"으로 축소됐을 뿐.

결론: §1.3을 3줄 사실관계(PG 내부 트랜잭셔널 / 현재는 PG가 중재 /
크로스 스토어 원자성 부재)로 재작성. #170은 "현재 상태는 PG가 결정한다"
원칙의 선행 사례로, 본 제안 §2.2와 같은 방향임을 명시.

## 2. PM 이의: "outbox로 뒤에서 뭉뚱그리는 것 + 불일치 장기화" → §2.2 재설계

이의 2건 모두 타당하다고 판단:

- 상시 coalescing은 평상시까지 "쓰기 N건 = 커밋 1개"로 뭉갬 — 피크 부하
  문제를 풀자고 평상시 히스토리 충실도를 영구 희생하는 과지불.
- 상시 비동기 git은 "데이터는 반영됐는데 히스토리에 안 보이는" 불일치를
  일상화함.

수정: **적응형 inline/deferred 드레인**으로 변경.

- 순서는 항상 PG-first (PG tx에 outbox 포함) — 스트레이 커밋 윈도우가
  구조적으로 소멸 (round-01 §1.3 문제의 해소이기도 함).
- 평상시: lane 처리자가 같은 요청 안에서 git 커밋까지 인라인 완료 —
  쓰기 1건 = 커밋 1개, 커밋 해시 동기 반환, 히스토리 지연 0. 사용자
  관점에서 현행과 동일.
- 압력 시(백로그 임계 초과/git 장애)에만 deferred + coalescing. 불일치
  지속 시간 ≈ 폭주 지속 시간 + 수 초 (level-based라 path당 1커밋으로
  백로그가 닫힘). 아카이버 장애는 `/health` 백로그 + 알림으로 가시화.
- deferred 구간 UX 보완: 히스토리 조회에 outbox/이벤트 기반 "아카이브
  대기 중" 가상 엔트리 오버레이.
- 새 비용으로 "두 모드의 복잡도"(전환 조건·응답 스키마 분기)를 §2.2
  비용 목록에 추가. 미결 #1은 "폭주 시 배치 허용 여부"로 범위 축소.
